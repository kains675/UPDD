## 4. v0.7 Entry Criterion Re-definition

### 4.1 Why Re-definition Is Needed

The v0.7 entry criterion was set during the v0.6.7 era as a **working assumption** (informal, not formally documented). Phase α evidence has invalidated this assumption.

**Working assumption during v0.6.7 era**:
> "Cp4 ΔΔG within reasonable range of Magotti -1.4/-3.0 anchor"
> (informal consensus, not specifically documented)

**Why this assumption is now problematic**:

1. **Methodology-specific implicit claim**: The criterion implicitly assumed PBSA + ff14SB could reach experimental quantitative accuracy. Phase α post-charge-fix evidence shows this is methodology-insufficient for Cp4 (CI95 excludes both -1.4 and -3.0).

2. **Methodologically conditioned result conflated with quantitative match**: Pre-patch Cp4 X.A UNCONDITIONAL appeared to satisfy the criterion, but Section 3 analysis shows this was a methodology-conditioned result, not a robust accuracy indicator.

3. **Single-system focus**: Original assumption was Cp4-only, ignoring multi-system generalization (1EBP_MTR13 success demonstrates path-validity beyond Cp4).

4. **Conflates architecture and accuracy**: Assumption lumped "infrastructure ready" with "quantitative match", which Phase α showed are independent axes requiring separate criteria.

### 4.1.1 UPDD Capability Levels — Honest Progression

UPDD's scientific capabilities are stratified into three levels. The current v0.7 release is explicitly Level 1; Levels 2 and 3 are roadmap targets, not present claims.

**Level 1 — Decision Support (current scope, v0.7)**
- Architectural integrity verified (Σq=0 across all production XMLs, audit infrastructure operational)
- Sign-significance demonstrated on at least one ncAA-modified peptide system
- σ statistics bounded within Genheden 2015 strict regime (σ_w < 10 kcal/mol)
- ΔΔG outputs serve as **decision support** for ranking ncAA variants in iterative design cycles, not as quantitative experimental predictions
- Honest acknowledgment of methodology ceilings (PBSA + ff14SB cannot match Magotti for Cp4 post-charge-fix)

**Level 2 — Cross-System Reproducibility (v0.7.1 / Phase β target)**
- Sign-significance reproduces on ≥3 systems spanning multiple ncAA classes
- σ_btwn vs σ_w decomposition pattern verified across systems
- Cross-class generalization demonstrated (PR-6 N-methyl overlay enables NML/NMA validation)
- WT control bit-identical established as routine reproducibility benchmark
- Output: comparative ranking confidence improves from single-system to multi-system

**Level 3 — Quantitative Match (v0.8+ / Phase γ target, partial expected)**
- Cp4 ΔΔG within field-comparable distance of Magotti -1.4/-3.0 anchor
- Khoury full RESP charges integrated; FF mixing validated at scale
- Cross-method agreement (e.g., FEP, alchemical) on at least one system
- Honest acknowledgment that Khoury R²=0.388 ceiling makes full Level 3 unlikely without Phase γ' refinements (entropy, polarization, or QM/MM)
- Output: predictive (not just decision support) within stated method's accuracy envelope

**Phase α' note**: Phase α' = post-Phase α deferred work expanding ncAA class coverage (PR-6/7/8 N-methyl, PRO, phospho overlays). Distinct from Phase β multi-system validation — Phase α' addresses *class coverage* for the same MTR-style architectural fix, while Phase β addresses *system-level reproducibility*.

### 4.1.2 UPDD Pipeline Architecture (Stage-aware)

UPDD operates as a 5-stage pipeline. Stage 4 (evaluation) is the primary contribution; Stages 1-3 integrate established generation tools, and Stage 5 closes the iterative loop with experimental feedback.

**Stage 1 — Sequence Input Parser**
- Parses user-supplied parent sequence + ncAA modification specification
- Validates against ncAA registry (current: 25+ ncAA entries, MTR-class verified, N-methyl class pending PR-6)
- Output: canonical design specification consumed by Stage 2

**Stage 2 — Generation (RFdiffusion + ProteinMPNN)**
- Backbone scaffolding (RFdiffusion) — established external tool
- Sequence design (ProteinMPNN) — established external tool
- UPDD does not modify these tools; integration is via standardized I/O contracts

**Stage 3 — Structure Prediction (AF2)**
- AlphaFold2 monomer/multimer prediction for designed sequences
- pLDDT / pTM filtering at user-configurable thresholds
- Output: 3D structure(s) consumed by Stage 4

**Stage 4 — Evaluation (UPDD primary contribution)**
- ncAA parameterization (parameterize_ncaa.py + ncaa_registry.py) with class-specific overlays
- MD equilibration + production (OpenMM, ff14SB protein + GAFF2/amb14SB-patched ncAA)
- QM/MM single-point energetics (gpu4pyscf, B3-style partitioning, R-15/R-16 charge guards)
- MM-PBSA / MM-GBSA scoring with σ_btwn vs σ_w decomposition
- Output: ΔΔG ranking + ensemble quality diagnostics

**Stage 5 — Iterative Feedback**
- Experimental wet-lab measurements (binding affinity, structural validation)
- Feedback ingested into Cycle 1+ design specification (Stage 1 input refinement)
- v0.7 supports Stage 5 as a documented protocol; automated wet-lab integration is a v0.9+ target

- Cycle 0 → wet lab K_d → best validated → Cycle 1 WT

### 4.2 Three Candidate Definitions Considered

#### 4.2.1 Process-Oriented vs Outcome-Oriented Distinction ⭐

Before evaluating specific definitions, a fundamental distinction:

**Process-oriented criteria** ("Did we set up the methodology correctly?"):
- Architectural correctness (charge integrity, audit infrastructure, cache cleanup)
- Verification protocol functioning (multi-layer error detection)
- Sign-significance demonstrated (signal-to-noise quality)
- Methodology setup quality assessment

**Outcome-oriented criteria** ("Does the methodology give right answers?"):
- Quantitative match to experimental anchors
- Cross-method agreement (e.g., FEP, SDM)
- Field-comparable accuracy benchmarks
- Methodology output quality assessment

**This distinction matters** because:
- Process can be verified independently of outcome
- Outcome can be measured against process-correct baseline
- Conflating the two produces false confidence (pre-patch X.A) or false despair (post-patch X.B)
- Versioning should reflect this separation

**v0.7 = process-oriented release**
**v0.8/0.9 = outcome-oriented releases**

This separation enables honest progress measurement and is rare in computational chemistry tooling (most projects conflate the axes).

### 4.6 v0.7 Verification Protocol Evidence ⭐

The Phase α + Option C process exhibited **two halt-and-recovery events** that validate the verification protocol design:

#### 4.6.1 Step 5.5 Cache Indirection Incident (Cron #41 catch)

**Event**: 2026-04-27 18:30 KST  
**Detection mechanism**: Automated cron sanity check  
**What it caught**: Step 6+7 v1 outputs bit-identical to pre-patch baseline (PBSA electrostatics-heavy → impossible if charges differ)  
**Root cause**: Cache indirection at scripts/run_mmpbsa.py L718-722 [pre-Option-C reference] (manifest xml_path → /tmp/calib_params/ stale 6-day-old XML)  
**Recovery**: Option A 4-location sync (cache + _md_input + manifests + clean _postpatch dirs)  
**Without this catch**: Step 6+7 v1 results would have been interpreted as "PBSA dominant, charge fix minor" — false conclusion would have propagated to Paper 1.

#### 4.6.2 Option C v1 Halt (Coder pre-merge gate catch)

**Event**: 2026-04-28 ~00:50 KST  
**Detection mechanism**: Coder pre-merge BREAK audit  
**What it caught**: Dispatch pattern `f"{ncaa_code}_gaff2.xml"` produced 10 BREAK candidates (NML manifests pointed to MLE_gaff2.xml local files due to xml_resname alias)  
**Root cause**: Dispatch text used narrower pattern than SciVal V2 §3.3 recommendation (glob *_gaff2.xml)  
**Recovery**: 1-line fix `xml_resname = manifest.get("xml_resname", ncaa_code)` for alias support  
**Without this halt**: Production code would have been modified to query non-existent file paths, breaking 10/37 manifests silently or with cryptic errors.
**Recovery outcome (Option C v2)**:
- 1-line fix: `xml_resname = manifest.get("xml_resname", ncaa_code)` for alias support
- Re-dispatch: 2026-04-28 ~01:08 KST
- Completion: 2026-04-28 ~01:20 KST (12 min wall, well under 30-45 min budget)
- Pre-merge audit v2: 0/37 BREAK ✅
- New tests: T1-T7 + legacy-compat = 8/8 PASS
- Regression suite: 117 pass / 9 skip / 0 fail (109 baseline + 8 new tests)
- Migration verdict: SAFE
